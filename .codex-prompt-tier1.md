## What to challenge

This is the THIRD codex pass on this codebase. Earlier passes already caught and fixed:
- Service-to-service auth (orchestrator SA needs roles/run.invoker on each peer; `_call_peer` fetches ID tokens with localhost bypass).
- incident_id idempotence (no wall-clock; sponsor filing_id from incident_id alone, not species).
- demo abuse (15s cooldown + asyncio.Lock).
- CORS browser break.
- event firehose cross-thread safety (threading.Lock on buffer + subscribers).
- Mapbox URL-restriction documented; default Firebase falls back to anonymous safely.

So go HARDER. Look for:

1. **P0 — security**:
   - Are the two new peer Dockerfiles + fast_api_app.py wired correctly for IAM (require Authorization header, validate audience, no anonymous bypass except localhost)?
   - Any token logging? Any leaks of the Authorization header or Bearer token into structured logs or events?
   - Funder/Neighbor peers — do they accept the SAME service-account auth pattern, or did the new code drift from the proven park_service/sponsor pattern?
   - PEER_ANCHORS coordinates hardcoded — any reserve PII or operational risk if a real reserve customer used this template?

2. **P0 — correctness / data integrity**:
   - 4-peer fanout via `asyncio.create_task` — what happens if one peer task raises an exception while the others succeed? Does the orchestrator's event firehose still emit the others' acks? Does it crash the request?
   - The auto-retry-on-echo path in `_call_peer` — could it accidentally double-file a peer receipt if the first call DID succeed but the heuristic mis-classifies the result as echoed? Walk through `_looks_echoed` against a successful response from each of the 4 peers.
   - `_send_once` extracted helper — does it preserve the request-id / message-id semantics A2A v0.3.0 wants for idempotence? If a retry happens, does the second message reuse or regenerate the messageId, and does the peer's `_processed_ids` cache match?
   - Funder reporter's `_VALID_PROGRAMS` — the orchestrator never validates the program name before sending. If Gemini hallucinates an off-list program, what's the user-visible failure mode in the Ops Center?
   - Neighbor park's `crossover_corridor` arg has NO validation in the orchestrator side — same question.

3. **P1 — reliability**:
   - 4 peers in parallel = 4 cold starts at first request. Are timeouts in `_send_once` / `_call_peer` long enough for 4 simultaneous Vertex AI Gemini cold starts? Or does poacher_truck systematically fail the first time it's fired in a fresh deploy?
   - The 15s cooldown is per-scenario, not global. Can a malicious client fire scenario1+scenario2+scenario3 in 100ms and bypass intent?
   - The auto-retry path emits a `peer_retry` event — does the Ops Center component panel handle that event type (or does it just stack ghost UI)?
   - Funder `narrative_hook` string contains user-controllable `species_affected` and `location` — XSS risk if the Ops Center renders it without escaping React's default escaping (which IS on by default, so this should be fine — confirm).
   - `_SEVERITY_TO_WINDOW_HOURS` in neighbor_park uses `obs_dt.replace(microsecond=0)` then adds a timedelta — if obs_dt is naive, the timedelta math is fine but the comparison to other times might surprise. Is the parsed obs_dt guaranteed tz-aware?

4. **P1 — UX / submission readiness**:
   - The Ops Center now has 4 active-peer cards in `IncidentPanel.tsx` — does the layout work on the 1280×720 video-recording viewport submission constraint? Or will the funder + neighbor cards push off-screen?
   - Demo cinema timing: 4 peers fan-out in parallel means all 4 arrows appear at once, not in narrative sequence. Is that the intended cinema, or should there be a small stagger to make the multi-agent coordination feel deliberate?
   - The `fanout` field on the event payload — does it match what `page.tsx` actually reads? Field name parity check.

5. **P2 — code quality**:
   - `_looks_echoed` heuristic is a string-match on missing fields + a marker. Is there a more principled detector (e.g. content-type heuristic on the A2A response) that would survive Gemini regressions better?
   - Both new peer agent.py files duplicate `_parse_ts`. Should this be in a shared `peers/_common/` module?
   - Severity-to-X dicts are duplicated across all 4 peers in different shapes. Pull to shared constants?
