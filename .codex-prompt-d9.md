## What to challenge

### P0 - security / correctness / privacy

1. **HTML injection via `render_evidence_html`.** The function reads ring-buffer event payloads and renders them into `<td>` cells. Every payload field passes through `html.escape()`. Walk through every interpolation point - is there a field type that bypasses escaping (lists rendered as strings, dicts coerced to repr, etc.)? Any path where a peer-controlled payload string lands raw in the HTML output is an XSS sink that auditors / judges could trigger.

2. **Sensitive data leakage in the bundle.** The Authorization Bearer tokens, ID tokens, or any other secret-bearing header MUST NOT appear in event payloads that the bundle walks. Check whether any tool_start/tool_end payload, a2a_request payload, or other firehose event could carry an Authorization header. If yes, the bundle leaks secrets to anyone who can hit `/demo/evidence/{id}`.

3. **Endpoint is unauthenticated.** `GET /demo/evidence/{incident_id}` and `/html` have no auth. Cloud Run service is currently `--allow-unauthenticated`. Anyone with an `incident_id` can pull the bundle. Acceptable for a hackathon demo (incident_ids are SHA-256 derived + hard to guess), but flag it.

4. **Path traversal via `incident_id`.** Does `bundle_incident` or `render_evidence_html` use `incident_id` anywhere that would let a malicious id (e.g., `../`) traverse filesystem, escape into a SQL query, or land in a regex unsafely? The current implementation appears to use it only as a dict key + string interpolation into output; verify nothing else.

5. **The chain_hash claim.** The bundle calls itself "tamper-evident" via SHA-256 of ordered event ids + timestamps. BUT - the ring buffer is in-memory + single-instance. The hash proves the bundle is internally consistent at the moment of generation; it does NOT prove the events were not lost / dropped before they reached the buffer. Is the "tamper-detection" framing overclaim?

### P1 - reliability / correctness

6. **Ring-buffer truncation handling.** `bundle_incident` sets `buffer_truncated=True` via the heuristic `len(evts) <= 2`. A real 4-peer incident produces 15+ events (1 incident_event + 4 a2a_request + 4 a2a_response + N tool_starts/ends from specialists). The heuristic catches the obvious case but: what if exactly half were evicted? The heuristic says false (>2 events), the bundle silently misses half the timeline. Does it matter? Is there a better signal (e.g., compare event count to expected baseline by scenario_id)?

7. **Time-window math fragility.** `duration_s` computation depends on the `ts` field being valid ISO-8601 (uses `replace("Z", "+00:00")`). If any event has a malformed ts (corrupted by a future code path), the fromisoformat raises -> the try/except returns duration_s=None. Silent data loss. Should this be logged?

8. **Specialist deduping.** `bundle_incident` partitions specialists by `agent in {"stream_watcher", "audio_agent", "species_id"}`. The Court-Evidence Agent ITSELF is named "court_evidence" - if it ever runs and emits tool_starts before bundling, those events would NOT be classified as specialists (correctly excluded) but ALSO would not be classified as peer_acks. They'd land only in the full timeline. Is that the intended split or a gap?

9. **HTML packet missing chain hash re-derivation instructions.** The packet says "the chain hash above can be re-derived by any auditor from the underlying event log." But it doesn't ship the event log alongside the HTML. Without the source events, the hash is unverifiable. Either ship the events JSON next to the HTML OR remove the unverifiable claim.

10. **No Content-Disposition on HTML response.** Auditors who click `/html` get inline-rendered HTML. They CAN print-to-PDF, but they cannot one-click-download as a file. Adding `Content-Disposition: attachment; filename="..."` would let them save the artifact directly. Is this a polish gap or a real auditor-workflow gap?

11. **`court_evidence_agent` model choice.** Uses gemini-2.5-flash. Per the project's prior codex finding (Park Service Flash echoed under concurrent load), Flash flakes on multi-arg tool calls. Court-Evidence has TWO tools each with single-arg incident_id - should be fine, but the ADK retry_options=attempts=3 is the only safeguard. Is gemini-2.5-pro safer for this agent given how legal-evidence-critical the output is?

12. **Race between `snapshot_for_incident` and concurrent `_broadcast`.** Acquires `_thread_lock` for the snapshot but then returns the iteration. If a `_broadcast` is in progress while the snapshot is in-flight, the iteration completes inside the lock so it's safe. Verify by walking the code path - any way the lock is released before iteration completes?

13. **Endpoint ordering vs FastAPI route registration.** The new evidence endpoints are added AT THE BOTTOM of `app/fast_api_app.py` after `if __name__ == "__main__"`. Wait - read the diff again: they're BEFORE the main block. But verify no other route already binds `/demo/evidence/{...}` and that FastAPI sees the new ones at import time.

14. **The `health` endpoint or any diagnostic does not yet expose the buffer depth for ops.** Operators can't quickly tell whether incidents are being evicted before bundling. Is this a gap or out-of-scope for hackathon?

### P2 - polish / consistency

15. **The `bundle_id` format inconsistency.** Park dispatches use `PSR-<digits>`. Sponsor uses `TNFD-<year>-<10hex>`. Funder uses `FUND-<year>-<10hex>`. Neighbor uses `MUTAID-<year>-<10hex>`. Court-Evidence now adds `CHN-<year>-<10hex>`. Consistent with sponsor/funder/neighbor format, inconsistent with park. Fine, but flag.

16. **Verbose tool docstrings.** Both `bundle_incident` and `render_evidence_html` have very long docstrings. ADK exposes these to the LLM as tool descriptions. Verbose descriptions are fine but should be terse enough that the LLM picks the right tool. Check.

17. **`_format_html` doesn't escape `bundle_id`.** Wait yes it does (`bid = html.escape(...)`). OK skip.
